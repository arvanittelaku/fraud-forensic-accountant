# SEO file generation

This document describes how `public/sitemap.xml` and `public/robots.txt` are produced, where the URL list comes from, and how the project keeps them in sync with the app.

## Overview

Both files are generated by a Node script rather than maintained by hand. That keeps every public URL aligned with the same inventory the verification tooling uses.

| Output | Path | Generator |
|--------|------|-----------|
| XML sitemap | `public/sitemap.xml` | `scripts/generate-seo.ts` |
| robots.txt | `public/robots.txt` | `scripts/generate-seo.ts` |

Next.js serves files in `public/` from the site root, so crawlers fetch:

- `https://www.fraudforensicaccountant.com/sitemap.xml`
- `https://www.fraudforensicaccountant.com/robots.txt`

The App Router does **not** use `src/app/sitemap.ts` or `src/app/robots.ts`; those were removed in favour of this pipeline.

## URL inventory (`buildPublicUrlInventory`)

The script imports `buildPublicUrlInventory` from `src/lib/seo/publicUrlInventory.ts`. That function assembles all paths that should appear in search, then maps them to absolute URLs with the canonical host `https://www.fraudforensicaccountant.com` (from `SITE_URL` in `src/lib/site.ts`).

The inventory merges:

1. **Static app routes** — `APP_STATIC_PATHS`: marketing and content pages (home, services hub, who-we-help, fraud/case type hubs, guides hub, contact, etc.).
2. **Service pages** — `/services/:id` for each entry in `src/data/services.ts`.
3. **Fraud type pages** — `/fraud-types/:slug` from `src/data/fraud-types.ts`.
4. **Case type pages** — `/case-types/:slug` from `src/data/case-types.ts`.
5. **Guide pages** — `/guides/:slug` from `src/data/guides.ts`.

Excluded from the inventory (noindex / not for normal indexing):

- `/thank-you`
- `/privacy`
- `/terms`

The combined list is deduplicated, sorted, and exposed as `allUrls` (absolute) and `allPaths` (pathname only).

## Sitemap generation (`scripts/generate-seo.ts`)

For each URL in `inventory.allUrls`, the script emits a standard sitemaps.org `<url>` entry with:

- `<loc>` — The absolute URL.
- `<lastmod>` — The calendar date when generation runs (YYYY-MM-DD in UTC).
- `<changefreq>` — Heuristic by path (`getSitemapChangeFreq`).
- `<priority>` — Heuristic by path (`getSitemapPriority`).

The full document is written to `public/sitemap.xml`. The `public/` directory is created if needed.

## robots.txt generation (`scripts/generate-seo.ts`)

`robots.txt` is a fixed template in the same script (`renderRobots()`), not derived from the URL inventory. It:

- Allows all crawlers on `/` by default.
- Disallows `/thank-you` and `/api/`.
- Declares the sitemap location: `Sitemap: https://www.fraudforensicaccountant.com/sitemap.xml`.

The file is written to `public/robots.txt`.

## Commands

From the repo root:

```bash
npm run seo:generate   # Regenerate sitemap.xml + robots.txt
npm run seo:verify     # Fail if sitemap is missing URLs vs inventory
```

`seo:generate` runs automatically before `npm run build`.

Run `seo:generate` after:

- Adding a path to `APP_STATIC_PATHS`
- Adding a service, fraud type, case type, or guide slug in data files
- Changing robots rules in `scripts/generate-seo.ts`
- Before a release when you want fresh `lastmod` dates

## Continuous integration

`.github/workflows/seo-checks.yml` runs on pull requests:

1. `npm ci`
2. `npm run seo:generate`
3. `npm run seo:verify`

## Operational notes

- Canonical host is `https://www.fraudforensicaccountant.com` in `src/lib/site.ts` and in the robots `Sitemap:` line. If the primary domain changes, update both and regenerate.
- New static pages must be added to `APP_STATIC_PATHS` or `seo:verify` will fail after the next generate.
- New dynamic sections need slugs in the relevant `src/data/*.ts` file; the inventory picks them up automatically.
